### Activity와 Fragment의 역사

안드로이드 앱의 UI는 **Activity**와 **Fragment**라는 두 가지 기본적이고 필수적인 구성 요소로 이루어집니다. 

하지만 이 둘이 처음부터 함께 존재했던 것은 아닙니다. 안드로이드 초기에는 **Activity**만 존재했고, **Fragment**는 나중에 **Activity**의 단점을 보완하기 위해 등장한 개념입니다.

 **🤔 그렇다면 왜 Fragment가 필요해졌을까요?**

스마트폰 초기에는 화면에 표시할 UI가 지금처럼 복잡하지 않았습니다. 하지만 스마트폰의 화면 크기가 다양해지고, 태블릿 같은 대화면 기기들이 등장하면서 문제가 발생하기 시작했습니다. 

하나의 **Activity**만으로 모든 화면을 구성하기에는 너무 복잡하고 비효율적이었습니다.

![image.png](attachment:d8320252-8af5-4a56-bf7e-5a3a957e7551:image.png)

예를 들어, 위의 하단 슬라이드 바처럼 여러 화면에서 공통으로 재사용하는 위젯이 있을 때, 이를 **Activity**마다 개별적으로 구현하는 것은 상당히 비효율적입니다. 

이럴 때 **Fragment**는 재사용이 가능한 UI 컴포넌트로서, 다른 **Activity**에 **종속**되어 동작함으로써 복잡성을 줄여줍니다. 

그래서  `DialogFragment`라든지 `NavigationDrawer` 같은 요소들도 **Fragment** 단위로 **Activity**에 **종속**되어 작동됩니다. 

여기서 **종속**에 대한 자세한 이야기는 추후 `LifeCycle`을 다루는 다른 챕터에서 다루도록 하겠습니다.

![image.png](attachment:e4430f86-70d2-46c0-8169-b8ef5c3ae141:image.png)

거기에다, **Fragment**는 **Activity**에 비해 메모리 소모도 적기 때문에, 구글은 2018년 이후로 하나의 **Activity**를 중심으로 여러 **Fragment**를 전환하며 앱의 화면을 구성하는 방식인 **SAA(Single Activity Architecture)**를 권장하고 있습니다.  

### Activity의 생명주기

안드로이드 앱을 만들 때 가장 중요한 개념 중 하나가 바로 **생명주기(Lifecycle)**입니다.
앱의 화면이 켜지고, 잠시 가려지고, 완전히 종료될 때까지 시스템이 정해진 순서대로 특정 함수들을 실행하는데, 우리는 이 타이밍에 맞춰 필요한 코드를 작성해야 합니다.

![image.png](attachment:8a3cfc96-954c-476a-8c22-73d8f7f6b174:image.png)

액티비티는 앱의 **하나의 전체 화면**을 담당합니다.

- **`onCreate()`**: 화면이 처음 만들어질 때 (딱 한 번 실행). 레이아웃을 연결하고 초기 데이터를 세팅합니다.
- **`onStart()`**: 화면이 사용자에게 보이기 시작할 때.
- **`onResume()`**: 사용자와 화면이 직접 상호작용(터치 등)을 할 수 있는 완전한 활성 상태.
- **`onPause()`**: 전화가 오거나 팝업이 떠서 화면의 일부가 가려진 상태. 하던 작업을 잠시 멈춥니다.
- **`onStop()`**: 홈 버튼을 누르는 등 화면이 완전히 가려진 상태.
- **`onDestroy()`**: 액티비티가 메모리에서 완전히 소멸될 때.

### Fragment의 생명주기

프래그먼트는 액티비티라는 큰 도화지 위에 올라가는 **조각 화면(부분 화면)**입니다. 액티비티에 종속되어 있기 때문에 생성과 소멸에 관한 단계가 조금 더 많습니다.

![image.png](attachment:e6d12971-6699-46e3-9611-f570603ff9fe:image.png)

- **`onAttach()`**: 프래그먼트가 액티비티에 딱 붙는 순간.
- **`onCreateView()`**: 프래그먼트의 **화면을 그리는 단계**. (xml 파일을 연결)
- **`onViewCreated()`**: 화면이 다 그려지고 난 직후. **버튼 클릭 이벤트나 데이터를 세팅하는 코드는 주로 여기에 작성합니다!**
- **`onDestroyView()`**: 프래그먼트의 화면(View)만 지워지는 단계.
- **`onSaveInstanceState()`**: 화면이 파괴되기 직전, 현재 상태(체크박스 체크 여부, 입력 중인 텍스트 등)를 **주머니(Bundle)**에 잠깐 담아두는 단계입니다.
- **`onViewStateRestored()`**: 다시 만들어진 화면에 **주머니에서 꺼낸 데이터를 복구**하는 단계입니다. 뷰(View)들의 상태가 모두 복원된 직후에 호출되므로, 안전하게 데이터를 다시 세팅할 수 있습니다.

### Activity와 Fragment의 사용 예시

이전 챕터에서 앱의 구성 요소에 대해 알아보셨다면, 4대 컴포넌트에 대해 들어보셨을 겁니다.

[ContentProvider](https://developer.android.com/reference/android/content/ContentProvider?hl=ko)`의 각 서브클래스의 [`<provider>`](https://developer.android.com/guide/topics/manifest/provider-element?hl=ko)

우리는 여기서 **Activity에** 집중해 볼 겁니다. 

**Activity**는 여전히 앱의 기본 화면 단위를 나타냅니다. **Activity**는 앱의 전체적인 틀과 맥락을 담당하며, 사용자가 앱을 조작할 때 가장 먼저 상호작용하는 진입점 역할을 합니다.

🤔그렇다면 **Fragment**는 언제 어디에 쓰일까요?

**Fragment**는 **Activity** 내에서 **재사용 가능한 작은 화면 조각**이라고 생각하면 이해하기 쉽습니다. 

 **Fragment**는 단독으로 존재할 수 없으며, 항상 **Activity**에 종속되어 동작합니다. 이러한 특성 덕분에 하나의 **Activity** 안에 여러 개의 **Fragment**를 담거나, 필요에 따라 **Fragment**를 교체하며 다양한 화면을 효율적으로 구성할 수 있습니다.

실제 코드를 통해 더 자세히 알아보도록 합시다.

```xml
<--activity_main.xml-->

    <com.google.android.material.bottomnavigation.BottomNavigationView
```

 위 코드에서  레이아웃은 크게 세 부분으로 나뉩니다.

1. **상단 바:** **ImageButton**과 **TextView**로 구성된 상단 UI
2. **중앙 Fragment 컨테이너:** `FragmentContainerView` 
3. **하단 내비게이션:** `BottomNavigationView` 

여기서 **Activity**가 담당하는 역할은 **상단 바와 하단 내비게이션 바처럼 모든 화면에 고정적으로 표시되는 요소**를 관리하는 것입니다.

반면, 중앙에 위치한 `FragmentContainerView`는 **Fragment**가 들어갈 **"그릇"** 역할을 합니다. 

![image.png](attachment:3e406642-0525-4828-b895-bbde0b1e758f:image.png)

여러분이 당근마켓이나 카카오톡에서 하단 탭을 누를 때마다 이 `FragmentContainerView`의 내용물, 즉 **Fragment가 동적으로 바뀌는 것**입니다.  

### BottomNavigationView란?

`BottomNavigationView`는 화면 하단에 여러 탭 아이템을 표시하여 사용자가 쉽게 화면을 전환할 수 있도록 돕는 위젯입니다. 보통 3~5개의 탭을 포함하며, 각 탭은 **Fragment**와 연결되어 있습니다.

![image.png](attachment:19b88bd6-2a11-4a74-b7b1-be76101cb1a5:image.png)

![image.png](attachment:0a06f936-6134-4447-9d40-d962df3a6f54:image.png)

위 이미지에서 아래의 표시된 부분이 `BottomNavigationView` 으로, 각 탭을 누를 때마다 탭에 알맞은 Fragment를 연결해줍니다.

**그렇다면 `BottomNavigationView`는 어떻게 그 형태와 구조를 정의할까요?** 

`BottomNavigationView`는 **`menu`** 파일을 통해 그 모양과 구조를 정의합니다.

- **menu 파일 정의 방법**
    1. `res` 디렉토리에서 `Android Resource Directory`를 선택합니다.
        
        ![image.png](attachment:eba78170-76eb-4425-9d28-973cadfbd256:image.png)
        
    2. Resource type을 `menu` 로 설정하고 추가합니다.
        
        ![image.png](attachment:3211bf24-bb02-4583-8850-f137645a304f:image.png)
        
    3. `menu` 디렉토리에서 `Menu Resource File`을 정의합니다.
        
        ![image.png](attachment:717715d5-830b-4efc-88a9-52c64157c1a3:image.png)
        

```xml

<menu xmlns:android="http://schemas.android.com/apk/res/android">
    <item
        android:id="@+id/homeFragment"
        android:icon="@drawable/ic_menu_home2"
        android:title="홈" />
    <item
        android:id="@+id/diaryFragment"
        android:icon="@drawable/ic_menu_diary2"
        android:title="일기쓰기" />
    <item
        android:id="@+id/calendarFragment"
        android:icon="@drawable/ic_menu_calendar2"
        android:title="달력" />
    <item
        android:id="@+id/friendFragment"
        android:icon="@drawable/ic_menu_friend2"
        android:title="친구" />
    <item
        android:id="@+id/mypageFragment"
        android:icon="@drawable/ic_menu_mypage2"
        android:title="마이페이지" />

```

```xml
<!--위의 activity_main.xml 중 bottomNav 코드 일부-->

<com.google.android.material.bottomnavigation.BottomNavigationView
        android:id="@+id/main_bnv"
        ...
        app:menu="@menu/bottom_nav_menu"
        ...
        />
```

이렇게 `menu` 파일에 아이템을 정의하고, `BottomNavigationView` 의 `app: menu` 와 연결하면, 각 아이템이 하단에 표시됩니다. 하지만 위는 단순히 **UI 측면에서의 연결**만을 다룬 것이기 때문에, 실제 앱처럼 터치 시 화면 전환 이벤트를 두려면 로직 즉 **Kotlin 파일을 수정**해야 합니다.

### FragmentManager를 통한 BottomNavigationView 전환

`FragmentManager`는 **Fragment**를 직접 제어하는 클래스입니다. 

`BottomNavigationView`의 아이템이 클릭될 때, `FragmentManager`를 사용하여 `FragmentContainerView`의 내용물을 수동으로 교체하는 방식입니다.

```kotlin
<!--MainActivity.kt-->

package com.example.umc_9th

import android.os.Bundle
import androidx.activity.enableEdgeToEdge
import androidx.appcompat.app.AppCompatActivity
import androidx.core.view.ViewCompat
import androidx.core.view.WindowInsetsCompat
import com.example.umc_9th.databinding.ActivityMainBinding

class MainActivity : AppCompatActivity() {

    private lateinit var binding: ActivityMainBinding

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        enableEdgeToEdge()
        binding = ActivityMainBinding.inflate(layoutInflater)
        setContentView(binding.root)

        supportFragmentManager.beginTransaction()
            .replace(R.id.main_bnv, BlankFragment())
            .commit()

				//BottomNavigationView를 눌렀을 때 Fragment 변경하기
        binding.mainBnv.setOnItemSelectedListener { item ->
            when (item.itemId){

                //매인 화면
                R.id.homeFragment -> {
                    supportFragmentManager.beginTransaction()
                        .replace(R.id.main_fragmentContainer, HomeFragment())
                        .commit()
                    true
                }

                //일기 작성 화면
                R.id.diaryFragment -> {
                    supportFragmentManager.beginTransaction()
                        .replace(R.id.main_fragmentContainer, DiaryFragment())
                        .commit()
                    true
                }

                //일기 히스토리 화면
                R.id.calendarFragment -> {
                    supportFragmentManager.beginTransaction()
                        .replace(R.id.main_fragmentContainer, CalendarFragment())
                        .commit()
                    true
                }

                //친구 화면
                R.id.friendFragment -> {
                    supportFragmentManager.beginTransaction()
                        .replace(R.id.main_fragmentContainer, FriendFragment())
                        .commit()
                    true
                }

                //마이페이지 화면
                R.id.mypageFragment -> {
                    supportFragmentManager.beginTransaction()
                        .replace(R.id.main_fragmentContainer, MypageFragment())
                        .commit()
                    true
                }
                else -> false
            }
        }
    }
}
```

먼저 **onCreate** 메서드에서 `BottomNavigationView` 객체를 가져옵니다. 그 후, `setOnItemSelectedListener`를 사용하여 각 탭 아이템 클릭 이벤트를 처리합니다.

그 후, **when** 문을 이용해 클릭된 아이템의 id에 따라 다른 **Fragment**를 교체하는 `replaceFragment` 함수를 호출합니다.

`replaceFragment` 함수는 `supportFragmentManager`를 통해 **Fragment**를 **트랜잭션(`beginTransaction`)으로 교체(`replace`)**하고 `commit`하여 변경 사항을 적용합니다.

### 안드로이드의 화면 전환: Intent & Navigation

앞서 `FragmentManager`를 통해 수동으로 화면(Fragment)을 갈아끼우는 방법을 알아봤습니다. 직관적이지만, 화면이 많아지면 관리가 까다로워진다는 한계가 있었죠.

안드로이드 앱은 여러 화면 사이 오가며, 사용자와 상호작용 합니다. 이 화면 전환은 크게 **두 가지 방식**으로 이루어집니다.

- **Activity 간 이동**: 앱의 독립적인 화면 단위인 **Activity** 사이를 이동할 때 사용합니다.
- **Fragment 간 이동**: 하나의 **Activity** 안에서 UI의 일부를 바꾸는 **Fragment** 사이를 이동할 때 사용합니다.

![image.png](attachment:f4379940-3aa2-469d-85e7-3ceed99191af:image.png)

**Activity**는 **`Intent`**를 이용해 전환 및 정보를 주고 받습니다.

**`Intent`**는 일종의 **매세지 객체**로, Android System에게 “나 이거 하고 싶어” 라는 내용을 담습니다.

```kotlin
//MainActivity.kt 중 일부
import android.content.Intent
...

class MainActivity : AppCompatActivity() {

    private lateinit var binding: ActivityMainBinding

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        enableEdgeToEdge()
        binding = ActivityMainBinding.inflate(layoutInflater)
        setContentView(binding.root)
				
				...
				//버튼 클릭 시 BlankActivity로 이동
				binding.myBtn.setOnclickListener{
						
						//Intent 객체 생성 : 현재(this)에서 BlankActivity로 이동할래요!
						val intent = Intent(this, BlankActivity::class.java)
						//Intent 실행
						startActivity(intent)
				}

    }
}
```

위처럼 **`Intent`** 객체를 생성해, **현재 Activity와 목표 Activity를 정의**하고, `startActivity()` 함수 안에 **`intent`** 객체를 전달하여 화면을 전환합니다.

그런데, 앱을 개발하다 보면, 화면을 전환하면서 **이전 화면의 특정 데이터를 다음 화면으로 전달**해야 하는 경우가 많습니다.

예를 들어, 1주차 미션에서, 각 버튼을 누를 때마다 어떤 감정을 선택했는지에 대한 정보를 다음 화면에서 보여주고 싶을 수 있습니다.  

이때 `Bundle`이라는 객체를 사용합니다. `Bundle`은 키(**key**)와 값(**value**)으로 구성된 묶음으로, 다양한 종류의 데이터를 담아 `Intent`에 실어 보낼 수 있습니다.

ex. MainActivity → BlankActivity로 이동 및 데이터 전달

```kotlin
//MainActivity.kt
...

//버튼 클릭 시 BlankActivity로 이동
				binding.myBtn.setOnclickListener{
						//Intent 객체 생성 : 현재(this)에서 BlankActivity로 이동할래요!
						val intent = Intent(this, BlankActivity::class.java)
						
						//NEW- ["emotion" : "Happy"] 삽입
						intent.putExtra("emotion", "Happy")
						
						//Intent 실행
						startActivity(intent)
				}
```

이렇게 `Intent`에 `.putExtra()` 함수를 사용하여 데이터를 담아 보내고,  이동한 **Activity**에서 `Intent`를 통해 `Bundle`을 꺼내고, **Key**값을 이용해 값을 얻을 수 있습니다.

```kotlin
//BlankActivity.kt
...
class BlankActivity : AppCompatActivity() {

    private lateinit var binding: ActivityBlankBinding

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        enableEdgeToEdge()
        binding = ActivityBlankBinding.inflate(layoutInflater)
        setContentView(binding.root)
				
				//Intent 꺼내기
				val emotion = intent.getStringExtra("emotion")
				
				//가져온 정보를 이용해 textView를 띄
				binding.tv2.text = emotion

    }
}
```

### **Fragment 화면 전환**

**Fragment**는 `BottomNavigationView` 전환에서 살펴봤듯이, **`FragmentManager`**와 **`NavGraph`** 두 가지 방법으로 화면을 전환할 수 있습니다.

1. **FragmentManager 이용**

`FragmentManager`는 개발자가 **Fragment의 전환 과정을 직접 제어**하는 방식입니다. 이 방법은 특정 위젯이 어떤 `Fragment`를 보여줄지, 그리고 어떤 옵션을 적용할지 세밀하게 정의할 수 있습니다.

- **장점:** **Fragment**의 생성, 추가, 제거 과정을 직접 제어하기 때문에 **세밀한 커스터마이징**이 가능합니다.
- **단점:** 코드가 길고 복잡해지며, 화면이 많아질수록 관리하기가 어려워집니다.

```kotlin
//HomeFragment.kt 일부

...

binding.myBtn.setOnClickListener {
    // 새로운 Fragment 인스턴스 생성
    val newFragment = DiaryFragment()
    
    // Fragment 교체 시작 및 실행
    supportFragmentManager.beginTransaction() 
        .replace(R.id.main_fragmentContainer, newFragment)
        .addToBackStack(null) // 이전 Fragment로 돌아갈 수 있도록 스택에 추가
        .commit()
}

```

1. **NavGraph 이용**

`NavGraph`는 **화면 이동 경로를 미리 정의**하여 **Fragment 전환**을 자동화하는 방식입니다.

 `NavGraph` 파일에 모든 화면(`Fragment`)과 이동 경로(`action`)를 미리 정의한 뒤, 코드에서는 **`NavController`** 객체를 찾아 이동할 `Fragment`의 ID를 호출하기만 하면 됩니다. 

그러면 나머지 전환 과정은 시스템이 자동으로 처리합니다.

```xml
<!--navGraph.xml-->

<?xml version="1.0" encoding="utf-8"?>
<navigation xmlns:android="http://schemas.android.com/apk/res/android"
    xmlns:app="http://schemas.android.com/apk/res-auto"
    xmlns:tools="http://schemas.android.com/tools"
    android:id="@+id/nav_graph"
    app:startDestination="@id/homeFragment">

    <fragment
        android:id="@+id/homeFragment"
        android:name="com.example.umc_9th.HomeFragment"
        android:label="홈"
        tools:layout="@layout/fragment_home">

        <!-- Home -> Diary로 이동하는 action 추가 -->
        <action
            android:id="@+id/action_homeFragment_to_diaryFragment"
            app:destination="@id/diaryFragment" />
    </fragment>

    <fragment
        android:id="@+id/diaryFragment"
        android:name="com.example.umc_9th.DiaryFragment"
        android:label="일기쓰기"
        tools:layout="@layout/fragment_diary" />

    ...

</navigation>
```

`<action>`에서 정의하는 **목적지 `app:destination`**은 이동할 목표 **Fragment의 ID**로, 미리 `<fragment>` 태그로 정의되어 있어야 합니다

```kotlin
//HomeFragment.kt 일부

...

binding.myBtn.setOnClickListener {
		//navController를 찾아서 action의 ID만 전
		val action = HomeFragmentDirections.actionHomeFragmentToDiaryFragment()
		findNavController().navigate(action)
		
    // findNavController().navigate(R.id.action_homeFragment_to_diaryFragment)
    // 위 주석처럼 간단하게 작성해도 상관없으나 아래에서 설명드릴 safe args에 대비하여
    // action과 같은 간단한 변수로 분리하는게 코드가 더 깔끔합니다 :)
}
```

![image.png](attachment:4c9dcc3e-365d-4a1d-bc93-e5a736e726e3:image.png)

`NavGraph`는 `Fragment` 전환 로직이 XML에 정의되어 있어 코드가 **간결**해지고, 안드로이드 스튜디오의 Navigation Editor를 통해 화면 흐름을 **한눈에 파악**할 수 있습니다.

### Safe Args: 안전하게 데이터를 배달하는 가이드

**Safe Args**는 NavGraph에 정의한 내용을 바탕으로 **데이터 전달용 클래스를 자동으로 만들어주는 플러그인**입니다. 플러그인 설정부터 배워보겠습니다!

1. toml 설정

```jsx
[versions]
# navigation 버전과 맞춰주세요
navigation = "2.8.0" 

[plugins]
# Safe Args 플러그인 추가
androidx-navigation-safeargs = { id = "androidx.navigation.safeargs.kotlin", version.ref = "navigation" }
```

1. Project 단위 gradle

```jsx
plugins {
    // ... 기존 플러그인들
    alias(libs.plugins.androidx.navigation.safeargs) apply false
}
```

1. App Module 단위 gradle

```jsx
plugins {
    alias(libs.plugins.androidx.navigation.safeargs)
}
```

gradle sync가 끝나면 이제 safe args를 사용할 수 있습니다!

먼저 어떤 데이터를 보낼지 XML에서 미리 약속해야 합니다. `Home`에서 `Diary`로 일기 제목(title)을 보낸다고 가정해 보겠습니다.

```jsx
<fragment
    android:id="@+id/diaryFragment"
    android:name="com.example.umc_9th.DiaryFragment"
    android:label="일기쓰기">
    
    <argument
        android:name="diaryTitle"
        app:argType="string"
        android:defaultValue="제목 없음" />
        
</fragment>
```

반드시 기억해야 될 내용! 데이터를 받는 쪽에 argument 태그를 붙이셔야 합니다!

</aside>

위처럼 코드가 작성되었다면

`HomeFragment`의 코드는 아래처럼 변경할 수 있습니다.

```jsx
binding.myBtn.setOnClickListener {
    // 1. 전달할 데이터가 담긴 Action 객체 생성
    val action = HomeFragmentDirections.actionHomeFragmentToDiaryFragment(
        diaryTitle = "오늘의 공부 일기" // 정의한 인자에 값 넣기
    )
    
    // 2. 생성한 action을 navigate에 전달
    findNavController().navigate(action)
}
```

데이터는 어떻게 받고 어떻게 사용하는지 보여드리겠습니다.

```jsx
// DiaryFragment

// 1. 전달받은 인자를 담을 변수 선언
private val args: DiaryFragmentArgs by navArgs()

override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
    super.onViewCreated(view, savedInstanceState)
    
    // 2. args 객체에서 꺼내서 사용하기 (타입이 이미 String으로 지정되어 있음!)
    val title = args.diaryTitle
    binding.textViewTitle.text = title
}
```

여기서 **`by`**는 어떤 역할을 하는 걸까요?

**`by`**는 **“이 변수의 초기화나 데이터 관리 로직을 다른 객체(위임체)에게 맡기겠다"**는 선언입니다.

즉 args라는 변수를 DiaryFragmentArgs 객체로 만드나 구체적인 시스템 작업은 navArgs에 위임하겠다는 뜻이 됩니다.

추후에 `by viewModels()` 나 값이 필요할 때 초기화하는 `by lazy` 등을 자주 만나게 될 거니 미리 알아두시면 이해하기 편하실 겁니다!
